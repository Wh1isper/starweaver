# Starweaver Claw Parity Matrix

This matrix cross-checks every reviewed Claw reference capability against the proposed `starweaver-claw` implementation on `starweaver-python`.

Reference snapshot: `refs/ya-mono` commit `e4012ce72a617285ec6e58bbe820fd87f787033a`.

Status meanings:

- `Implement now`: feasible with current `starweaver-python` plus product-owned code.
- `Implement now, product-owned`: feasible, but intentionally remains outside generic Starweaver SDK/runtime.
- `Implement now with compatibility adapter`: feasible, but requires explicit product adapter for API/schema/transport compatibility.
- `Non-blocking Starweaver addition`: product can implement now; later Starweaver generalization is proposed in `../alignment/08-starweaver-claw-sdk-additions.md`.

## Blocking Assessment

No reviewed reference feature is architecturally blocked if `starweaver-claw` owns the compatibility adapters and product orchestration listed in this matrix. Each feature remains gated by the validation tests below before claiming parity.

The most expensive product-owned implementations are:

1. Reusable Docker sandbox lifecycle.
2. Live SSE replay/tail and AGUI compatibility adapter.
3. HITL durable HTTP/bridge workflow.
4. Product async background task lifecycle.
5. Workflows, memory, and agency dispatchers.

These are product features in the reference implementation; keeping them product-owned is the safest route to full parity.

## Verification Rule

A feature should be considered aligned only when both checks pass:

1. Product compatibility test: the ported reference behavior test passes against `starweaver-claw`.
2. Starweaver evidence test: canonical Starweaver records exist for the underlying run/session/stream where the feature executes an agent run.
